Phoenix建立索引

8 创建索引

8.1 创建全局索引

首先创建一张测试表如下：

0: jdbc:phoenix:127.0.0.1:2181> create table user(id integer primary key,name varchar,sex varchar,phone varchar);

在该表的 name 字段上建立 desc 索引，这个索引还包含了 phone 字段的值。

0: jdbc:phoenix:127.0.0.1:2181> create index if not exists idx_user on user(name desc) include(phone);

添加两条测试数据：

0: jdbc:phoenix:127.0.0.1:2181> upsert into user values(1,'zhang','male','18711111111');
1 row affected (0.525 seconds)
0: jdbc:phoenix:127.0.0.1:2181> upsert into user values(2,'li','male','13974000000');

包含索引字段的查询：

0: jdbc:phoenix:127.0.0.1:2181> select name,phone from user where name = 'zhang';
+--------+--------------+
|  NAME  |    PHONE     |
+--------+--------------+
| zhang  | 18711111111  |
+--------+--------------+
1 row selected (0.068 seconds)

如果查询语句中还需要知道其他字段的值，例如：

0: jdbc:phoenix:127.0.0.1:2181> select * from user where name = 'zhang'; 
+-----+--------+-------+--------------+
| ID  |  NAME  |  SEX  |    PHONE     |
+-----+--------+-------+--------------+
| 1   | zhang  | male  | 18711111111  |
+-----+--------+-------+--------------+
1 row selected (0.155 seconds)

可以发现，和没有索引的速度差不多，因为该索引中没有包含 sex 这个字段。

另外需要注意的是：主键不需要索引，查询也非常快，这是因为 HBase 已经对 RowKey 建立了索引。

可能你会想要查看自己创建的索引，可以使用 !indexes tablename，例如：

0: jdbc:phoenix:127.0.0.1:2181> !indexes user
+------------+--------------+-------------+-------------+------------------+-------------+-------+----------------+
| TABLE_CAT  | TABLE_SCHEM  | TABLE_NAME  | NON_UNIQUE  | INDEX_QUALIFIER  | INDEX_NAME  | TYPE  | ORDINAL_POSITI |
+------------+--------------+-------------+-------------+------------------+-------------+-------+----------------+
|            |              | USER        | true        |                  | IDX_USER    | 3     | 1              |
|            |              | USER        | true        |                  | IDX_USER    | 3     | 2              |
|            |              | USER        | true        |                  | IDX_USER    | 3     | 3              |
+------------+--------------+-------------+-------------+------------------+-------------+-------+----------------+

在 HBase 的 命令行下输入 list 命令同样也可以看到刚刚创建的索引：

这样我们就可以确认，全局索引确实是在HBase中重新创建了一张表。

通过以上测试可以看出全局的设计方式，会单独写一张索引表，列族为 include 字段，RowKey 的设计方位是:

  • 二级索引字段1 +二级索引字段2(复合索引) + 原表rowkey；
  • 查询的时候，会直接定位到索引表，通过 RowKey查到位置，然后从索引表中带出数据；
  • 因为建立索引的时候还要多写一份include字段，读的时候直接从索引表定位并读出信息。所以这种表的应用场景定位是写的慢，读得快。

如果想要删除索引可以使用：

drop index if exists IDX_USER on USER;

8.2 创建本地索引

创建表与本地索引：

0: jdbc:phoenix:127.0.0.1:2181> create table user_local(id varchar primary key,name varchar,sex varchar,phone varchar);

0: jdbc:phoenix:127.0.0.1:2181> create local index local_idx_user on user_local(name);

添加测试数据，测试：

0: jdbc:phoenix:127.0.0.1:2181> create local index local_idx_user on user_local(name);
0: jdbc:phoenix:127.0.0.1:2181> upsert into user_local values('1','zhang','male','18117111111');
0: jdbc:phoenix:127.0.0.1:2181> upsert into user_local values('2','wang','female','18917111111');

查询：

0: jdbc:phoenix:127.0.0.1:2181> select name,phone from user_local where name = 'zhang';
+--------+--------------+
|  NAME  |    PHONE     |
+--------+--------------+
| zhang  | 18117111111  |
+--------+--------------+
1 row selected (0.077 seconds)
0: jdbc:phoenix:127.0.0.1:2181> select name,phone,sex from user_local where name = 'zhang';
+--------+--------------+-------+
|  NAME  |    PHONE     |  SEX  |
+--------+--------------+-------+
| zhang  | 18117111111  | male  |
+--------+--------------+-------+
1 row selected (0.068 seconds)

通过 hbase shell 查看 LOCAL_USER 表：

hbase(main):050:0> scan 'USER_LOCAL'
ROW                                         COLUMN+CELL                                                                                                                  
 \x00\x00wang\x002                          column=L#0:\x00\x00\x00\x00, timestamp=1566462124189, value=\x00\x00\x00\x00                                                 
 \x00\x00zhang\x001                         column=L#0:\x00\x00\x00\x00, timestamp=1566462083288, value=\x00\x00\x00\x00

 1                                          column=0:\x80\x0B, timestamp=1566462083288, value=zhang                                                                      
 1                                          column=0:\x80\x0C, timestamp=1566462083288, value=male                                                                       
 1                                          column=0:\x80\x0D, timestamp=1566462083288, value=18117111111                                                                
 2                                          column=0:\x00\x00\x00\x00, timestamp=1566462124189, value=x                                                                  
 2                                          column=0:\x80\x0B, timestamp=1566462124189, value=wang                                                                       
 2                                          column=0:\x80\x0C, timestamp=1566462124189, value=female                                                                     
 2                                          column=0:\x80\x0D, timestamp=1566462124189, value=18917111111

通过上述测试可以发现：

  • 第一，第二行的RowKey和下面的有一些不同，这两行就是索引数据；
  • 本地索引的设计方式，索引数据直接写在原表RowKey中，列族不写任何实际信息；
  • 可以发现本地索引的 start key 都是 "\x00"，这样做的目的是保证索引数据和原数据在一个 Region 上，定位到二级索引后根据原 RowKey 就可以很快在本 Region 上获取到其它信息，减少网络开销和检索成本；
  • 查询的时候，会在不同的region里面分别对二级索引字段定位，查到原rowkey后在本region上获取到其它信息；
  • 因为这种索引设计方式只写索引数据，省了不少空间的占用，根据索引信息拿到原rowkey后再根据rowkey到原数据里面获取其它信息。所以这种表的应用场景定位是写的快，读得慢。
